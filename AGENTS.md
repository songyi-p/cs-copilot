# CS Copilot 저장소 작업 지침

## 프로젝트 개요

- 이 프로젝트는 Next.js 16과 React 19를 사용하는 고객센터 운영용 백오피스 데모입니다.
- TypeScript와 Tailwind CSS를 사용합니다.
- 데모용 정적 데이터는 `src/data`에 저장합니다.
- 클라이언트의 데모 상태는 `localStorage`에 저장합니다.
- 현재 버전에서는 실제 고객 메시지를 발송하거나 실제 환불을 실행하지 않습니다.

## 주요 경로

- `src/app`: Next.js 페이지와 서버 라우트입니다.
- `src/components/dashboard`: 고객센터 대시보드 UI입니다.
- `src/utils/types.ts`: 공통 도메인 타입입니다.
- `src/utils/lib.ts`: 정책 검색 및 도메인 유틸리티입니다.
- `src/data/policies`: 원본 정책 문서입니다.
- `src/data/policy-search-index.json`: 검색용 정책 섹션입니다.
- `docs/data-model.md`: 도메인 데이터 모델 문서입니다.

## Git 브랜치 규칙

- 새로운 브랜치는 `main`에서 생성합니다.
- 신규 제품 기능 및 동작 변경에는 `feature/<kebab-case>`를 사용합니다.
- 동작 변경이 없는 리팩터링에는 `refactoring/<kebab-case>`를 사용합니다.
- UI 버그를 포함한 버그 수정에는 `bug/<kebab-case>`를 사용합니다.
- 버그 수정이 아닌 UI 디자인 변경 및 화면 개발에는 `ui/<kebab-case>`를 사용합니다.
- 이 저장소에서는 `codex/` 접두사를 사용하지 않습니다.
- 사용자의 관련 없는 변경사항은 그대로 보존합니다.
- 명시적으로 요청받지 않았다면 커밋, 푸시 또는 Pull Request 생성을 하지 않습니다.

## 개발 규칙

- TypeScript를 사용하고 `any` 타입 사용을 피합니다.
- 공통 타입은 `src/utils/types.ts`에 정의된 타입을 재사용합니다.
- 소스 코드 import에는 `@/` 별칭을 사용합니다.
- 기존 컴포넌트 구조와 Tailwind 스타일 작성 방식을 따릅니다.
- 가능하면 도메인 로직을 화면 컴포넌트와 분리합니다.
- 기존 기술 스택으로 명확하게 구현할 수 있다면 새로운 의존성을 추가하지 않습니다.

## LLM 연동 규칙

- LLM 호출은 서버 코드에서만 수행합니다.
- API 키나 LLM 공급자 인증 정보를 클라이언트 컴포넌트에 노출하지 않습니다.
- LLM에는 고객 문의, 관련 주문 정보, 검색된 정책 섹션만 전달합니다.
- 고객 프로필, 담당자 정보, 처리 이력 또는 관계없는 개인정보를 전달하지 않습니다.
- LLM 응답은 신뢰할 수 없는 입력으로 간주하고 실행 시점에 검증합니다.
- LLM에 전달한 정책에 실제로 존재하는 정책 근거만 허용합니다.
- 신뢰도가 `low`이면 자동 권장 처리안 대신 `담당자 이관 권장`을 표시합니다.
- LLM 호출이 실패하면 실패 상태를 표시하며, 하드코딩된 문구를 AI가 생성한 결과인 것처럼 대신 표시하지 않습니다.

## 검증 방법

- 구현을 변경한 후 `npm run build`를 실행합니다.
- 작업 완료 전에 TypeScript 오류를 확인합니다.
- UI를 변경했다면 로딩, 성공, 빈 결과, 오류 및 비활성화 상태를 확인합니다.
- 클라이언트 요청에 LLM 전달 허용 목록 이외의 데이터가 포함되지 않았는지 확인합니다.
- 실제로 검증 명령을 실행하지 않았다면 검증에 성공했다고 말하지 않습니다.

## 작업 완료 기준

- 요청한 기능이 처음부터 끝까지 연결되어 동작해야 합니다.
- 오류 및 로딩 상태가 처리되어야 합니다.
- 인증 정보나 빌드 결과물을 Git에 추가하지 않습니다.
- 기존 티켓 임시 저장, 승인, 이관 및 `localStorage` 흐름이 계속 동작해야 합니다.
- API 계약이나 데이터 모델이 변경되면 관련 문서도 함께 수정합니다.
