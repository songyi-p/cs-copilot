# CS Copilot 저장소 작업 지침

## 프로젝트 개요

- 이 프로젝트는 Next.js 16과 React 19를 사용하는 고객센터 운영용 백오피스 데모입니다.
- TypeScript와 Tailwind CSS를 사용합니다.
- 데모용 정적 데이터는 `src/data`에 저장합니다.
- 클라이언트의 데모 상태는 `localStorage`에 저장합니다.
- 현재 버전에서는 실제 고객 메시지를 발송하거나 실제 환불을 실행하지 않습니다.

## 주요 경로

- `src/app`: Next.js 페이지와 서버 라우트입니다.
- `src/components/dashboard`: 고객센터 대시보드 UI입니다.
- `src/hooks`: 화면 상태와 TanStack Query 기반 서버 상태를 관리하는 클라이언트 훅입니다.
- `src/server`: OpenAI SDK를 사용하는 서버 전용 LLM 로직입니다.
- `src/utils/schemas.ts`: AI 요청·응답의 공통 Zod 스키마와 타입입니다.
- `src/utils/req.ts`: Axios 기반 클라이언트 API 요청 함수입니다.
- `src/utils/types.ts`: 공통 도메인 타입입니다.
- `src/utils/lib.ts`: 정책 검색, 저장 상태 병합, 스타일 조합 유틸리티입니다.
- `src/data/policies`: 원본 정책 문서입니다.
- `src/data/policy-search-index.json`: 검색용 정책 섹션입니다.
- `docs/data-model.md`: 도메인 데이터 모델 문서입니다.

## Git 브랜치 규칙

- 새로운 브랜치는 `main`에서 생성합니다.
- 신규 제품 기능 및 동작 변경에는 `feature/<kebab-case>`를 사용합니다.
- 동작 변경이 없는 리팩터링에는 `refactoring/<kebab-case>`를 사용합니다.
- UI 버그를 포함한 버그 수정에는 `bug/<kebab-case>`를 사용합니다.
- 버그 수정이 아닌 UI 디자인 변경 및 화면 개발에는 `ui/<kebab-case>`를 사용합니다.
- 이 저장소에서는 `codex/` 접두사를 사용하지 않습니다.
- 사용자의 관련 없는 변경사항은 그대로 보존합니다.
- 명시적으로 요청받지 않았다면 커밋, 푸시 또는 Pull Request 생성을 하지 않습니다.
- Pull Request의 제목과 본문은 한글로 작성합니다.

## 개발 규칙

- TypeScript를 사용하고 `any` 타입 사용을 피합니다.
- 공통 타입은 `src/utils/types.ts`에 정의된 타입을 재사용합니다.
- 소스 코드 import에는 `@/` 별칭을 사용합니다.
- 기존 컴포넌트 구조와 Tailwind 스타일 작성 방식을 따릅니다.
- 가능하면 도메인 로직을 화면 컴포넌트와 분리합니다.
- 변수와 함수 이름은 보통 2~4개 단어 안에서 역할이 드러나게 작성하고, 같은 문맥이 반복되면 불필요한 접두사와 중복 단어를 제거합니다.
- `req`, `res`, `max`, `min`, `len`, `ref`, `info`, `id`, `config`처럼 팀에서 의미를 바로 이해할 수 있는 축약어는 긴 식별자를 줄이는 데 사용할 수 있습니다. 한 글자 이름이나 도메인 의미가 사라지는 임의 축약은 피합니다.
- 이벤트 핸들러는 `handle*`를 반복해서 붙이기보다 `saveDraft`, `approveTicket`, `closeDialog`처럼 동작을 직접 나타냅니다.
- 페이지 파일은 라우트 진입점과 화면 조립에 집중합니다. 여러 컴포넌트가 공유하는 상태, 파생값, 저장 및 업무 동작은 해당 화면의 `use*` 훅으로 분리합니다.
- 훅 파일은 훅 이름과 동일한 camelCase를 사용합니다. 예: `useDashboard.ts`, `useAiSuggestion.ts`.
- 두 곳 이상에서 같은 의미와 스타일로 반복되는 버튼, 배지, 다이얼로그 같은 UI는 `src/components/common`의 공통 컴포넌트로 추출합니다.
- 공통 컴포넌트는 도메인 상태를 직접 해석하지 않고 `variant`, `tone`, `size`와 같은 작은 표현 API를 제공합니다. 도메인 값과 UI 표현의 매핑은 사용하는 쪽이나 상수 모듈에 둡니다.
- 컴포넌트는 화면 표현과 짧은 UI 상태만 소유합니다. `localStorage`, API 요청 조립, 처리 이력 생성처럼 업무 흐름에 해당하는 로직은 훅이나 유틸리티로 분리합니다.
- `src/utils`는 `req.ts`, `types.ts`, `formatters.ts`, `constants.ts`, `schemas.ts`, `lib.ts`를 기본 구성으로 사용합니다. 기능이 작고 강하게 연관된 유틸리티는 새 파일을 만들기보다 기존 모듈에 함께 둡니다.
- 확정성, 작업 효율성, 유지보수성 측면에서 기대 효과가 더 크다면 새로운 의존성 사용을 권장합니다.
- 새로운 의존성을 추가할 때는 충분히 검증되고 현재 유지보수되고 있는 안전한 라이브러리인지 확인합니다.
- 역할이 중복되는 라이브러리는 피하고, 각 의존성의 도입 목적을 명확히 합니다.

## LLM 연동 규칙

- LLM 호출은 서버 코드에서만 수행합니다.
- API 키나 LLM 공급자 인증 정보를 클라이언트 컴포넌트에 노출하지 않습니다.
- `.env.local`과 인증서, API 키 등 비밀정보가 포함될 수 있는 로컬 파일은 사용자가 명시적으로 허용하지 않는 한 열람, 검색, 수정 또는 설정 여부 확인을 하지 않습니다.
- 환경변수 변경이 필요하면 기본적으로 `.env.example`과 문서, 코드 기본값만 수정하고 실제 로컬 환경변수 변경은 사용자에게 안내합니다.
- LLM에는 고객 문의, 관련 주문 정보, 검색된 정책 섹션만 전달합니다.
- 고객 프로필, 담당자 정보, 처리 이력 또는 관계없는 개인정보를 전달하지 않습니다.
- LLM 응답은 신뢰할 수 없는 입력으로 간주하고 실행 시점에 검증합니다.
- OpenAI Structured Outputs와 Zod 스키마를 함께 사용해 응답 구조와 TypeScript 타입을 일치시킵니다.
- 클라이언트의 LLM 서버 상태는 TanStack Query로 관리하고 Axios를 API 전송 계층으로 사용합니다.
- LLM에 전달한 정책에 실제로 존재하는 정책 근거만 허용합니다.
- 신뢰도가 `low`이면 자동 권장 처리안 대신 `담당자 이관 권장`을 표시합니다.
- LLM 호출이 실패하면 실패 상태를 표시하며, 하드코딩된 문구를 AI가 생성한 결과인 것처럼 대신 표시하지 않습니다.

## 검증 방법

- 작업을 기능과 목적에 따라 논리적인 단위로 나누고, 각 단위가 끝나면 커밋합니다.
- 각 커밋 전에는 변경 파일과 diff를 확인하되, 매 커밋마다 전체 빌드를 실행하지 않습니다.
- 전체 구현이 끝난 후 `npm run build`를 한 번 실행합니다.
- 위험도가 높은 변경이나 독립적인 검증이 필요한 경우에만 중간 빌드를 실행합니다.
- 최종 빌드가 실패하면 원인을 수정하고 필요한 검증을 다시 실행합니다.
- 작업 완료 전에 TypeScript 오류를 확인합니다.
- UI를 변경했다면 로딩, 성공, 빈 결과, 오류 및 비활성화 상태를 확인합니다.
- 클라이언트 요청에 LLM 전달 허용 목록 이외의 데이터가 포함되지 않았는지 확인합니다.
- 실제로 검증 명령을 실행하지 않았다면 검증에 성공했다고 말하지 않습니다.

## 작업 완료 기준

- 요청한 기능이 처음부터 끝까지 연결되어 동작해야 합니다.
- 오류 및 로딩 상태가 처리되어야 합니다.
- 인증 정보나 빌드 결과물을 Git에 추가하지 않습니다.
- 기존 티켓 임시 저장, 승인, 이관 및 `localStorage` 흐름이 계속 동작해야 합니다.
- API 계약이나 데이터 모델이 변경되면 관련 문서도 함께 수정합니다.
